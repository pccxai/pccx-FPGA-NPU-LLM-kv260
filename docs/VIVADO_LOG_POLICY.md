# Vivado Log Policy

This repository keeps Vivado source inputs and curated build conclusions
under git. Raw Vivado runtime logs are local build artifacts and stay
gitignored.

## Tracked

Commit these files when they explain or reproduce the build flow:

| File type | Examples | Notes |
|-----------|----------|-------|
| Build scripts | `hw/vivado/build.sh`, `hw/vivado/*.tcl` | Source inputs for the flow. |
| Documentation | `hw/vivado/README.md`, `docs/TIMING_EVIDENCE.md`, this file | Human-written policy, checklist, and result interpretation. |
| Curated summaries | `docs/evidence/**/summary.txt`, `docs/evidence/**/*.md` | Small, reviewed summaries are acceptable when they capture release or PR evidence. |
| Blocker notes | `docs/evidence/**/blocker.txt` | Use these for board, license, or tool blockers that explain missing build evidence. |

Curated evidence must be short enough for review and must name the Vivado
version, command, target branch or commit, and final result. Copy only the
specific warning/error/report lines needed to support the conclusion.

## Gitignored

Do not commit raw files generated by Vivado batch runs, project creation,
synthesis, implementation, crash handling, or GUI sessions:

| File type | Examples | Reason |
|-----------|----------|--------|
| Logs | `vivado.log`, `hw/build/vivado_synth.log`, `*.backup.log` | Large, machine-local, and noisy across runs. |
| Journals | `vivado.jou`, `hw/build/vivado_impl.jou`, `*.backup.jou` | Command transcripts, not reviewed source. |
| Runtime state | `vivado_pid*.str`, `*.str`, `.Xil/` | Host/process-specific state. |
| Project/build trees | `hw/build/`, `*.runs/`, `*.cache/`, `*.gen/`, `*.hw/`, `*.sim/` | Rebuildable generated output. |
| Binary outputs | `*.dcp`, `*.bit`, `*.bin`, `*.hwh`, `*.xsa`, `*.xclbin` | Heavy generated artifacts; publish them outside git when needed. |
| Crash logs | `hs_err_pid*.log` | Host-specific tool crash output. |

The top-level `.gitignore` intentionally ignores `*.log`, `*.jou`,
`*.str`, `.Xil/`, and `hw/build/`, so Vivado logs should not appear in
normal `git status` output. Do not use `git add -f` for these files.

## PR Evidence Format

For a PR that ran Vivado, add a short note in the PR body or in a tracked
summary file:

```text
Vivado: 2025.2
Command: cd hw && PCCX_VIVADO_JOBS=1 ./vivado/build.sh synth
Commit: <short-sha>
Result: PASS or FAIL
Evidence: timing/utilization/DRC summary values, or the minimal blocker
```

If the build failed, capture the first actionable error and any relevant
tool/version context. Leave the raw `.log` and `.jou` files in the local
worktree or CI artifact store.
